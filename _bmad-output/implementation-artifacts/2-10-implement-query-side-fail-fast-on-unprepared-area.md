# Story 2.10: Implement query-side fail-fast on unprepared area (FR24)

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a user,
I want `steeproute` to exit 2 with an actionable error message pointing me at `steeproute-setup` when the query area isn't fully covered by any prepared cache entry,
so that I discover the missing prep immediately and know exactly what to run.

## Acceptance Criteria

1. New `cache.py::check_coverage(cache_root, query_area) -> PreparedData` resolves the query against the cache: it reads `index.json` (rebuilding via `rebuild_index` if the file is missing, empty, or unparseable), builds a `shapely` polygon for the query bbox (same WGS84 km→deg approximation used by `_bounds_geojson` so the geometry matches what setup wrote), and tests strict `shapely.contains` of each indexed area's polygon against the query polygon. If exactly one area contains the query, `read_entry` returns its `PreparedData`. If multiple contain the query, the smallest by `radius_km` wins (minimum graph-load cost). If none contain, `CacheNotFoundError` is raised with an actionable `user_message` — see ACs #3 / #4 for the two distinct cases.

2. `cli/query.py::cli` is wired end-to-end through `check_coverage`: after `validate_area_size` it builds an `Area`, calls `resolve_cache_root(cache_dir)`, then `check_coverage(cache_root, area)` — replacing today's stub `_ = (...)` flag-acknowledgement block. On success it acknowledges the returned `PreparedData` (the solver isn't wired yet — Epics 3+) and prints a one-line cache-hit summary on stdout (mirroring `cli/setup.py`'s `cache-hit` cue: `steeproute: cache-hit  cache_key_hash: <hex>`). `PreExecutionError` subclasses raised by `check_coverage` propagate to `run_entry_point` and produce exit 2 with the existing stderr formatting.

3. Empty / missing-index case: when no prepared entries exist (cache root absent, `areas/` empty, or `index.json` has zero entries), `check_coverage` raises `CacheNotFoundError` whose `user_message` is the *empty-cache* variant: `No prepared cache covers this area. Run: steeproute-setup --center <lat>,<lon> --radius <km> --dem-path <your DEM>` with `<lat>,<lon>` and `<km>` echoed from the query's own `--center`/`--radius` (so the suggested command is directly copy-pasteable).

4. Coverage-miss case (entries exist but none contain the query): `check_coverage` raises `CacheNotFoundError` whose `user_message` names the nearest prepared area(s) and suggests a smaller `--radius` actionable for re-running setup or query. "Nearest" is by great-circle distance between the query center and each candidate entry's center; "smaller radius" is the largest `radius_km` for which strict containment by the nearest entry would hold (or "narrow your `--center`/`--radius` to fall within …" if the query's center is itself outside every prepared area). One nearest area is sufficient; ties are broken by sorted `cache_key_hash`. `detail` (verbose-only) lists every prepared area's center + radius for diagnostic purposes.

5. Multi-containment case: when ≥2 indexed areas strictly contain the query, the entry with the smallest `radius_km` is selected (ties broken by sorted `cache_key_hash`); the returned `PreparedData` is loaded via `read_entry` so cache-corruption is surfaced through the existing `CacheCorruptedError → exit 2` path, not a new code path. No error or warning is emitted on multi-containment — picking the smallest is the documented contract.

6. OSM-age warning lift on the query-side cache-hit path: the `_emit_osm_age_warning(manifest, threshold_days, now)` helper landed in `cli/setup.py` (Story 2.9) is consumed by `cli/query.py` against the resolved `PreparedData.manifest` after `check_coverage` returns. The shared home is `cli/_shared.py::emit_osm_age_warning` (lifted unchanged from `cli/setup.py`, with the setup-side call site updated to import from the new home). The query-side warning message suggests `steeproute-setup --force-refresh` (the query CLI itself has no `--force-refresh` flag); the helper signature is unchanged but the warning-string template moves to a module-level constant in `cli/_shared.py` so both CLIs can compose the right "what to re-run" wording without duplicating the rest. `cli/query.py` gains the `osm_age_warn_days_option` decorator (today absent from its stack — verified against the current file) so `--osm-age-warn-days` is parseable on both CLIs with the same 90-day default.

7. New `tests/e2e/test_coverage_check.py` exercises the three real-cache scenarios end-to-end using real `steeproute-setup` invocations as test setup (no mocked coverage data, no hand-written `index.json`): (a) `test_no_prepared_cache_exits_2_with_setup_command` runs `steeproute` against an empty `--cache-dir` and asserts exit 2 + the empty-cache stderr line + the literal `steeproute-setup --center` prefix in the suggested command; (b) `test_partial_coverage_exits_2_with_nearest_area_message` seeds one small prepared area via `setup_cli`, then runs `steeproute` against a query whose bbox pokes outside it, asserts exit 2 + nearest-area message + the smaller-radius suggestion; (c) `test_multi_containment_picks_smallest_radius` seeds two concentric setup runs (radius 5 km and 10 km), runs `steeproute` against a 2-km-radius query at the same center, asserts exit 0 + stdout `cache-hit` + `entry: …` points at the 5 km entry's `cache_key_hash`. The fixture-driven `osm_load` patch from `test_steeproute_setup.py` (`_osm_load_from_fixture`) is reused for the setup-seeding step.

8. Opportunistic `index.json` rebuild closes deferred-work D1 from Story 2.7's review: `check_coverage` calls `rebuild_index(cache_root)` whenever (a) the index file is missing, (b) it parses but is empty while `areas/` contains valid entries, or (c) its parse fails. An integration test in `tests/integration/test_cache.py` (or new `test_cache_coverage.py` if cleaner) constructs a real cache with two `write_entry` calls, then deletes `index.json` between them simulating the AC's edge case, then asserts `check_coverage` against a query contained by one of the entries rebuilds the index and returns the right `PreparedData`.

9. Unit tests in `tests/unit/test_cache.py` (or new `test_cache_coverage.py`) pin the pure containment logic with hand-built `Manifest` + `IndexEntry`-equivalents (no I/O): three areas at different `radius_km` around the same center, a query strictly contained by all → smallest wins; a query strictly contained by exactly one → that one wins; a query contained by none → `CacheNotFoundError` with the AC #4 nearest-area message format; an empty entry list → `CacheNotFoundError` with the AC #3 empty-cache message format. Boundary semantics — `shapely.contains` is strict (a query polygon that touches but doesn't sit strictly inside the indexed polygon is NOT contained) — covered by an explicit parametrize case.

10. All four CI gates pass on Windows: `uv run ruff check`, `uv run ruff format --check`, `uv run basedpyright`, `uv run pytest --cov`. `cache.py` stays ≥95% (pure-logic floor per Architecture §Cat 11e); the new `cli/query.py` wiring is exercised by the e2e tests in AC #7 (CLI tier remains exempt from the 95% floor). No new runtime dependencies — `shapely` is already declared. Live OSM test (`pytest -m live`) is unaffected by this story but re-verified at the end as a smoke check.

## Dev Notes

- **Polygon shape and km→deg approximation.** Coverage check builds the query polygon the same way `cache.py::_bounds_geojson` builds the entry's bounds: an axis-aligned WGS84 bbox at the area's center, `radius_km` as the bbox half-side, with the simple equator approximation `1° lat ≈ 111 km` and `1° lon ≈ 111 km × cos(lat)` (Architecture §Cat 4a sidecar contract). Pull that math into a small `_area_to_polygon(area: Area) -> shapely.Polygon` helper in `cache.py` and use it from both `_bounds_geojson` (refactor: build the polygon, then emit GeoJSON from its `.exterior.coords`) and `check_coverage` so query and entry share one source of truth. A polygon-skew between the two would cause phantom misses on edge cases.

- **Why `check_coverage` reads `index.json` instead of walking `areas/*/manifest.json` directly.** Architecture §Cat 4 frames `index.json` as a coverage-lookup convenience file; walking the manifests at every query startup would pay the per-entry JSON-parse + schema-validation cost for every prepared area on every run. The index is a derived state that lists `cache_key_hash` + `area`, which is exactly what coverage needs. On the rare cases the index is missing or stale (the Story 2.7 D1 deferred KeyboardInterrupt window), `check_coverage` opportunistically calls `rebuild_index` and retries the read. This is the right home for the rebuild because (a) `check_coverage` is the first reader on the query path, and (b) the rebuild cost is amortized — once paid, subsequent queries hit the rebuilt index for free.

- **Why the `IndexEntry` parse lives next to `check_coverage`, not as a top-level dataclass.** The index schema is two fields per entry (`cache_key_hash`, `area`). Lifting it to a `@dataclass` for v1 is overkill — `check_coverage` parses the JSON into a list of `(cache_key_hash: str, area: Area)` tuples (or a small NamedTuple internal to `cache.py`) and works from there. If a third reader of the index materializes, promote it then.

- **`shapely.contains` semantics.** `Polygon.contains(other)` (DE-9IM mask `[T*****FF*]`) returns True iff no point of `other` lies outside `self`'s interior **or boundary** — meaning identical polygons return True (a polygon contains itself), and polygons whose edges coincide are admitted. A query polygon that pokes outside the indexed polygon on any side is NOT contained, which is what Architecture §Cat 4e's "strict containment" rule actually means in practice ("no out-of-bounds query coverage"). AC #9 has explicit parametrize cases for both the identical-bbox and pokes-outside boundaries so this contract is pinned in tests. *(Earlier drafts of this Dev Note cited the more restrictive `[T**FF*FF*]` mask — a no-boundary-touching variant. Shapely's actual `contains` uses the boundary-permissive mask above, which matches what the implementation and tests pin.)*

- **Smaller-radius suggestion math.** For the "nearest area" diagnostic in AC #4, the actionable "smaller `--radius`" suggestion is the largest value `r_new` such that a bbox of half-side `r_new` centered on the original query center fits strictly inside the nearest prepared area's bbox. Closed-form: `r_new = max(0, min(nearest.radius_km - |Δlat_km|, nearest.radius_km - |Δlon_km|))` where `Δlat_km` and `Δlon_km` are the center-to-center deltas in km (using the same approximation as the polygon builder). If `r_new <= 0`, the query center is outside the nearest prepared area entirely — fall back to "Narrow `--center` to within the prepared area at <lat>,<lon> or run `steeproute-setup` for your target area" rather than suggesting a non-positive radius.

- **OSM-age warning lift.** The helper landed in `cli/setup.py` (Story 2.9). Its docstring already noted "design `_emit_osm_age_warning` so it can be lifted (or imported) without modification." Moving it to `cli/_shared.py::emit_osm_age_warning` (rename: drop the module-private underscore since it's now cross-CLI) and updating `cli/setup.py`'s call site is the minimum-diff lift. Tests in `tests/unit/test_cli_setup.py` need their import target updated; consider extracting a generic `tests/unit/test_cli_shared.py::test_emit_osm_age_warning_*` if the test surface stays meaningful at the new home (or leave the tests where they are with adjusted imports — pick the smaller diff).

Full query-side summary (FR22) lands in Epic 4; Story 2.10's stdout cue is a one-liner placeholder, not the final summary format.

### Completion Notes List

**Design decisions worth review attention:**

1. **Shared `_area_to_polygon` helper.** Both `_bounds_geojson` (entry-side, persisted as `bounds.geojson`) and `check_coverage` (query-side, transient) now build their polygons via a single `_area_to_polygon(area: Area) -> shapely.Polygon` helper in `cache.py`. The refactor of `_bounds_geojson` keeps the on-disk shape byte-identical — `Polygon.exterior.coords` produces the same 5-vertex closed ring the hand-built code emitted — and `tests/unit/test_cache.py::test_bounds_geojson_geometry_and_properties_center_use_lon_lat_consistently` still passes unchanged. Sharing the geometry source means a polygon-skew between query and entry is structurally impossible.

2. **`shapely.contains` semantics.** Architecture §Cat 4e says "strict containment" but uses `shapely.contains` literally. `shapely.Polygon.contains(other)` follows DE-9IM `[T*****FF*]`: returns True when no point of `other` lies in the polygon's exterior — that means identical polygons return True, equal-edge touches from inside are True, and any out-of-bounds poke is False. This matches the user contract: a query that matches a prepared area exactly is a HIT (not a miss). Pinned explicitly by `test_area_to_polygon_containment_boundary_semantics` with four cases (identical, smaller-same-center, larger-same-center, shifted-out).
