## What worked

- **Lightweight review, consistently applied.** Every story 1.2 onward closed via inline lightweight review (config / scaffolding / small pure-Python diffs). Findings were structured (`D1`, `D2`, … with explicit dismiss/defer reasoning) rather than narrative. The pattern matched the kind of work — heavier `bmad-code-review` would have been ceremony for ceremony's sake on these surfaces. Worth keeping for similarly-scoped stories.
- **`deferred-work.md` is doing real work.** 6 items deferred from Story 1.1 → all resolved in Story 1.3. The 1.3 → Story 5.5 (coverage `omit` list) and 1.6 → first-`--radius`-consumer (negative-radius validation) handoffs are clearly traceable. This avoided "fix it now or forget it forever" pressure during review.
- **Findings hygiene was honest.** The recurring "N findings raised, all dismissed" pattern in [`sprint-status.yaml`](sprint-status.yaml) comments was *not* rubber-stamping — each dismissal cites a specific reason (false-positive, redundant, intent-marker, spec-sanctioned, etc.). Explicit dismissals mean future readers can re-evaluate.
- **Acceptance criteria stayed at outcome altitude across the whole epic.** Even the bloated stories had clean Given/When/Then ACs. The bloat was downstream of ACs, in Dev Notes — not in the contract layer.

## What we learned

### Story-file bloat drift, then a clean correction (the main lesson)

Story files grew bloated through 1.1–1.4, then snapped back at Story 1.5 and held lean through 1.7.

| Story | Lines | Dev Notes sections beyond template canon |
|---|---|---|
| 1.1 | 306 | (early scaffolding — many sections inherent) |
| 1.2 | 314 | Architecture/PRD alignment, Previous-story intelligence, Source tree, Implementation sketches, Scope boundaries, Anti-patterns, Verification commands |
| 1.3 | 439 | + "What the CI workflow looks like after this story" (~57 lines), "What `pyproject.toml` looks like" (~34 lines), Target repo tree |
| 1.4 | 413 | "Concrete implementation sketches" alone was ~99 lines |
| **1.5** | **190** | ✂️ Stripped back to template canon: Project Structure Notes, Testing standards, References |
| 1.6 | 162 | (held lean) |
| 1.7 | 162 | (held lean) |

The bloated subsections were all variants of the same anti-pattern: *pre-implementing the work in prose*. Target file contents, verification command transcripts, anti-pattern catalogues, source-tree screenshots. They pretended to be context but were really shadow-implementations the dev agent had to keep in sync with the actual code.

The correction at 1.5 was to stick to the BMAD `template.md` canon (Project Structure Notes / Testing standards / References) and let acceptance criteria carry the contract. ~100–200 lines is the right target band; 1.5/1.6/1.7 all landed there. This is now the standing rule for future stories.

### Other observations

- **Story 1.3 was authored ahead of 1.2 and rebased onto it later** ([sprint-status.yaml:60](sprint-status.yaml:60)). Decoupled deliverables can be parallelized when the dependency note is explicit. Worked here; available as a tool for future epics where stories are genuinely independent.
- **Lightweight review surfaced one real bug, deferred cleanly.** Story 1.6's negative-`--radius` silent-acceptance ([deferred-work.md:34-40](deferred-work.md:34)) is a genuine bug that was correctly out of scope for 1.6 and is properly parked at the consumer site. The lightweight review process is catching real issues, not just rubber-stamping.

### Practices to keep

- Lean Dev Notes (Project Structure / Testing standards / References + anything genuinely story-specific). Watch for any "what file X looks like after this story" subsection — that's the bloat warning sign.
- Lightweight inline review with structured findings (D1/D2… + explicit dismiss/defer rationale) for similarly-scoped diffs.
- `deferred-work.md` updated as part of close-out, not after.
- Story close-out commit + fast-forward to `main` from the worktree.

### Practices to reassess for Epic 2

- **Lightweight review may no longer be the default.** Epic 2 stories involve real logic (OSM ingestion + filtering, 2D smoothing/resampling, DEM sampling, cache key hashing, atomic cache writes, end-to-end orchestrator). Several of these are closer to "material algorithmic work" than "config scaffolding" — heavier `bmad-code-review` may fit. Decide per story rather than defaulting either way.
