# Story 2.9: Handle source-unavailable errors and emit OSM-age warnings

Status: done

## Story

As a user,
I want `steeproute-setup` to exit with a clear `error:` line and exit code 2 when the OSM or DEM source is unreachable, and to emit a stderr warning when a cache-hit entry's OSM extract date exceeds `--osm-age-warn-days`,
so that data-source issues are distinguishable from bugs and stale data is surfaced without blocking the happy path.

## Acceptance Criteria

1. Network or I/O failures inside `osm_load` (Overpass / `osmnx.graph_from_point`) surface as `DataSourceUnavailableError` whose `user_message` starts `OSM source unreachable` (identifying the OSM side). The original exception is wrapped via `raise … from exc` and its `repr` lands on the `detail` line so `--verbose` surfaces it. The boundary is the single `osmnx.graph_from_point(...)` call site in `pipeline/osm.py::osm_load` — wrap that call only; the upstream argument validation in `_validate_area` keeps its `BadCLIArgError` contract.

2. I/O failures inside `sample_elevation` at the `rasterio.open(dem_path)` site surface as `DataSourceUnavailableError` whose `user_message` starts `DEM source unreachable`, with the original `rasterio.RasterioIOError` / `OSError` wrapped via `raise … from exc` for `--verbose`. The wrap covers DEM-not-found, permission-denied, and corrupt-GeoTIFF cases (deferred-work DEF2 from Story 2.8). The existing `DEMCoverageError` paths inside `sample_elevation` (no-CRS, inverted bounds, out-of-bounds vertex, nodata sample) are NOT changed — coverage problems are categorically distinct from source-unavailable and keep their existing message + exit-2 mapping via the shared `PreExecutionError` base.

3. On a successful cache-hit in `steeproute-setup` (the read-only branch in `cli/setup.py::cli` where `read_entry` succeeds and `--force-refresh` is not set), compare `manifest.osm_extract_date` to current UTC time; if the delta exceeds `--osm-age-warn-days` (default 90, already plumbed through `cli/_shared.py::osm_age_warn_days_option`), emit a single `logging.warning(...)` on stderr identifying the entry's age in days and suggesting `--force-refresh`. The warning is non-blocking, non-fatal, and emitted before `_print_summary`. The check is a small pure helper (e.g. `cli/setup.py::_emit_osm_age_warning(manifest, threshold_days, now)`) so the unit test in AC #6 can drive it without booting the full CLI. `manifest.osm_extract_date` is the `"YYYY-MM-DDTHH:MM:SSZ"` string written by Story 2.8 via `provenance.iso8601_utc_now()`; parse it with `datetime.datetime.fromisoformat`. A malformed `osm_extract_date` (defensive — Story 2.7's `Manifest.from_dict` would have already raised `CacheCorruptedError`) skips the warning rather than crashing the hit path.

4. New `tests/e2e/test_source_unavailable.py::test_missing_dem_path` drives a real filesystem failure: `--dem-path <tmp_path>/nonexistent.tif` (with the rest of the args fixture-derived as in `test_steeproute_setup.py`). The current `dem_path.is_file()` guard in `cli/setup.py` is the one that fires today and produces `BadCLIArgError` — **this test must reach the `rasterio.RasterioIOError` path**, so use a file that exists but isn't a valid GeoTIFF (e.g. `tmp_path / "garbage.tif"` with arbitrary bytes), OR temporarily bypass the CLI's `is_file()` guard by patching it, OR (preferred) write a zero-byte `.tif` whose `Path.is_file()` returns True but which `rasterio.open` rejects. Assert: exit code 2, stderr starts `error: DEM source unreachable`, no traceback (`PreExecutionError` formatting from `run_entry_point`).

5. New `tests/e2e/test_source_unavailable.py::test_osm_network_failure` uses `monkeypatch` to make `osmnx.graph_from_point` raise `requests.ConnectionError(...)` (or equivalent — `requests.exceptions.ConnectionError` is what propagates in production). Mocking is required here because the test's purpose IS to validate the simulated failure handling, per epic spec. Assert: exit code 2, stderr starts `error: OSM source unreachable`, `--verbose` rerun surfaces the wrapped `ConnectionError` repr on the detail line.

6. Unit test in `tests/unit/test_cli_setup.py` (new file, or fold into an existing CLI-unit file — pick the smaller diff) drives `_emit_osm_age_warning` with a synthetic `Manifest` whose `osm_extract_date` is dated > 90 days ago, asserts a single `logging.warning(...)` is emitted on stderr (use `caplog`) naming the age in days and suggesting `--force-refresh`. A second parametrized case at exactly the threshold (age == 90 days, age == 89 days) pins the boundary semantics — pick one inclusive convention and document it in the helper's docstring. A third case with a fresh `osm_extract_date` (age = 0 days) asserts no warning.

7. e2e test `test_osm_age_warning_emitted_on_stale_cache_hit` (new, in `test_source_unavailable.py`) pre-seeds a cache entry via the existing `test_steeproute_setup.py` pattern (in-process `CliRunner` + `patch("steeproute.pipeline.osm_load", ...)`), monkey-patches `provenance.iso8601_utc_now` so the seeded manifest's `osm_extract_date` lands > 90 days in the past, then re-invokes setup without `--force-refresh` and asserts: exit code 0, stdout summary still shows `cache-hit`, stderr contains the OSM-age warning. This is the integration-tier evidence that the AC #3 wiring is reachable from the real CLI. The unit test in AC #6 is the boundary-case authority.

8. All four CI gates pass on Windows: `uv run ruff check`, `uv run ruff format --check`, `uv run basedpyright`, `uv run pytest --cov`. The new pipeline-side `try/except` blocks land in `pipeline/osm.py` (covered to 95% per Architecture §Cat 11e pure-logic floor) and `pipeline/dem.py` (already at 100% — keep it there); the new CLI-side helper in `cli/setup.py` is exercised by the unit test in AC #6 (the CLI tier remains exempt from the 95% floor, but the helper itself is pure-logic and should hit the bar via its dedicated unit test). No new runtime deps. Live OSM test (`pytest -m live`) re-verified at the end.

## Dev Notes

- **Exception scope for the OSM wrap.** `osmnx.graph_from_point` can raise a wide family: `requests.exceptions.ConnectionError`, `requests.exceptions.Timeout`, `requests.exceptions.HTTPError`, the bare `OSError` family, and osmnx's own `osmnx._errors.InsufficientResponseError` for empty-area responses. Catch `(OSError, Exception)` is too broad — it'd swallow programming errors. Catch `(requests.exceptions.RequestException, OSError)` covers the documented network failure modes plus permission/IO-tier issues without masking `ValueError` / `TypeError` from bad inputs. Document this decision inline. The `InsufficientResponseError` / empty-result case is **out of scope** — the orchestrator's `_assert_non_empty` already raises `PipelineContractError` with an actionable message (deferred-work finding 2 from Story 2.1 is already resolved by that guard; mark it resolved-elsewhere in the `deferred-work.md` close-out).

- **Exception scope for the DEM wrap.** `rasterio.open` raises `rasterio.errors.RasterioIOError` on file-not-found / corrupt / permission-denied. `RasterioIOError` is a subclass of `OSError`, so catching `OSError` would cover both, but naming `RasterioIOError` explicitly (and falling through to `OSError` for the truly-unexpected case) keeps the diagnostic clearer when reviewing tracebacks. The `is_file()` guard in `cli/setup.py` catches the common "user typoed the path" case earlier; the new wrap covers what slips past it (TOCTOU race, permission denied without `is_file()` returning False, corrupt header, network filesystem timeout).

- **Why the OSM-age check lives in `cli/setup.py` and not `cache.py`.** Architecture §Cat 4f frames the warning as user-facing CLI output, not a cache-layer invariant — `cache.py`'s job is "is this entry valid?" (it is; `osm_extract_date` is metadata, never part of the cache key). The warning is presentation-tier; reading the manifest's age and deciding to log lives at the CLI. The same helper will be needed in `cli/query.py` once Story 2.10 wires the query-side cache-hit path — design `_emit_osm_age_warning` so it can be lifted (or imported) without modification.

- **Test isolation for the OSM-age e2e test.** The seeded manifest's `osm_extract_date` is captured at `write_entry` time, so the patch must be active during the seed invocation, not the verification invocation. Pattern: `with patch("steeproute.provenance.iso8601_utc_now", return_value="2024-01-01T00:00:00Z"): _invoke_setup(...)` for the seed, then a plain re-invocation (no patch) drives the cache-hit path and triggers the warning against the real `datetime.datetime.now(UTC)`.

- **What this story does NOT do:**
  - **Query-side fail-fast / `check_coverage`** — Story 2.10.
  - **Setup-time DEM nodata-undeclared sanity check** (`0.0`-as-void with `nodata=None`) — out of scope; the production fixture has `nodata=None` and is fully covered. ### Completion Notes List

**Design decisions worth review attention:**

1. **OSM exception scope: `(requests.exceptions.RequestException, OSError)`.** `RequestException` is the base of every `requests` failure class (`ConnectionError`, `Timeout`, `HTTPError`, `SSLError`, ...). Catching that base covers the documented network failure modes osmnx propagates from its HTTP backend. `OSError` is a defensive add-on for truly low-level issues (network-filesystem disconnect, exhausted file descriptors). We deliberately do NOT catch `Exception` — programming errors (TypeError/ValueError from bad inputs) must keep their original tracebacks for diagnosis. The osmnx-specific `osmnx._errors.InsufficientResponseError` (empty-area response) is out of scope per the Dev Notes: it bubbles up as a non-empty graph with zero trail edges, and the orchestrator's `_assert_non_empty` guard already surfaces that with the right actionable message.

2. **DEM exception scope: `(rasterio.errors.RasterioIOError, OSError)`.** `RasterioIOError` is rasterio's primary "couldn't read the raster" class and is a subclass of `OSError`. Listing both is redundant but reads clearly in tracebacks. The existing `DEMCoverageError` paths inside the `with rasterio.open(...)` block (no-CRS, inverted bounds, out-of-bounds vertex, nodata sample) are deliberately preserved — they represent "DEM opened fine but the data is wrong shape for this area", categorically distinct from "DEM source unreachable" per Architecture §Cat 10. The structural shape needed a small refactor: `rasterio.open` is now called outside the `with` block (wrapped in try/except) and re-entered as `dataset_ctx as dataset`. This keeps the existing dataset-using code unchanged.

3. **Threshold semantics: strict `>` not `>=`.** `_emit_osm_age_warning` warns only when `age_days > threshold_days`, not `>=`. Rationale: a fresh `--osm-age-warn-days 90` install with an entry written today shouldn't warn on day 90 — that's the user's intentional cadence. The first warn-day is 91. Pinned by `test_emit_osm_age_warning_boundary_semantics` parametrized over `(89, 90, 90.5, 91)` days.

4. **`fromisoformat` + tz handling.** `provenance.iso8601_utc_now()` writes a literal `Z` suffix; Python 3.11+'s `datetime.fromisoformat` parses `Z` as UTC. A defensive `if extract_dt.tzinfo is None: extract_dt = extract_dt.replace(tzinfo=datetime.UTC)` handles the edge case where someone hand-edited the manifest to a naive ISO string (or a future schema migration drops the Z) so the `now - extract_dt` subtraction doesn't raise `TypeError: can't subtract offset-naive and offset-aware datetimes`.

5. **Malformed `osm_extract_date` is swallowed silently.** `Manifest.from_dict` (Story 2.7) already raises `CacheCorruptedError` on schema violations before this helper sees the manifest. Reaching the `ValueError` branch in `_emit_osm_age_warning` requires a user hand-editing the file mid-run. The age warning is auxiliary diagnostic information; losing it on a malformed date is acceptable. Crashing the cache-hit happy path because of it is not. Pinned by `test_emit_osm_age_warning_swallows_malformed_extract_date`.

6. **In-process exit-2 testing pattern.** The existing `test_steeproute_setup.py` covers SUCCESS paths via `CliRunner.invoke(setup_cli, ...)` and routes ERROR paths through `test_cli_smoke.py`'s subprocess tests. Story 2.9 needed both monkey-patching (in-process) AND exit-code assertions. The `_invoke_with_wrapper` helper bridges this: it calls `setup_cli.main(args=..., standalone_mode=False)`, catches `PreExecutionError`, and mirrors `run_entry_point`'s stderr-write + exit-code logic in the test layer. Side-effect: tests see the same `error: ...\n` format the binary writes, validating the user-facing contract not just the exception-type contract.

7. **The OSM-network-failure tests must patch `osmnx.graph_from_point`, not `pipeline.osm_load`.** Patching `pipeline.osm_load` directly bypasses the wrap entirely (the wrap is INSIDE `osm_load`'s body). The right target is the `osmnx.graph_from_point` call site — patching `steeproute.pipeline.osm.osmnx.graph_from_point` triggers the production try/except. This is documented inline so a future maintainer doesn't "simplify" by switching to the wrong target.

8. **The OSM-age cache-hit e2e test seeds via `iso8601_utc_now` patch, not by hand-writing a stale manifest.** Writing a manifest directly would couple the test to the cache schema; patching the source of truth for "now" during the seed step keeps the test future-proof against schema changes. The verification step intentionally re-invokes WITHOUT the patch so the warning fires against the real `datetime.datetime.now(UTC)`. Also asserts the cache-hit branch never reaches `osm_load` (passes `patch_osm=False`).

9. **Carry-forwards closed out.** Resolved-in-2.9: D1 from Story 2.1 (osmnx error handling) and DEF2 from Story 2.8 (DEM permission-denied). Resolved-elsewhere (marked in `deferred-work.md`): D2 from Story 2.1 (empty-graph result — Story 2.5's `_assert_non_empty` handles it) and D2 from Story 2.4 (NaN elevation absorption — Story 2.5's `_assert_finite_elevations` handles it).
