# steeproute end-to-end performance review

**Date:** 2026-07-24  
**Reviewer:** GPT-5.6-sol  
**Scope:** setup + query, preserving result quality and deterministic behavior  
**Reference commit:** `4380970`  
**Status:** investigation and POCs only; no production implementation

## Executive summary

The previous optimization passes were effective. On the current r20 workload, setup is no longer
dominated by elevation sampling, and the four-worker solver is no longer the only important query
cost. The largest remaining opportunities are object-graph churn and repeated derivation of immutable
graph state.

The strongest findings are:

1. **Three query orchestration changes produced a byte-identical end-to-end result and reduced
   CLI-reported wall-clock from 80.02 s to 67.33 s (−15.9%).** External process wall fell from
   90.82 s to 78.62 s and peak RSS from 2.67 GB to 2.05 GB. The changes were:
   - filter the freshly loaded operational graph in place instead of building a duplicate;
   - strip rendering-only attributes from the contracted graph in place and send it directly to
     workers instead of rebuilding a second “lean” graph;
   - compute validation's graph-wide reuse/distinctness invariants once, not once per route.

   Every one of the 20 output files (10 HTML + 10 JSON) was byte-for-byte identical between the
   baseline and POC.

2. **osmnx's largest-component implementation is the best setup-side CPU target.** On the cached
   91 MB r20 Overpass response, standard warm ingestion took 131.94 s, including 33.57 s in two
   largest-component calls. A behavior-equivalent POC traversed components once and removed rejected
   nodes from the owned graph in place. Those calls fell to 2.77 s combined and total ingestion to
   99.10 s (−24.9%). The real graph retained the same 131,793 nodes and 327,911 edges.

3. **Elevation sampling still has about 7.6 s of low-risk r20 headroom around the already-vectorized
   core.** Its input copy measured 5.39 s, and per-edge geometry extraction measured 3.76 s versus
   1.51 s with Shapely's bulk coordinate API, with bit-identical coordinates. The setup CLI owns the
   graph at this point, so a consuming path can avoid the copy.

4. **Fuse setup polyline smoothing and resampling.** On an r20-shaped graph, profiling attributed
   about 7.4 s of smoothing to building an intermediate NetworkX graph that resampling immediately
   flattens again. Keep the coordinates flat across both stages and build the graph once.

5. **The parallel solver's hard limit is worker state construction, not just the hot loop.** On r20,
   cProfile attributed 12.36 s to adjacency construction and 11.35 s to 100k GRASP iterations
   (profiled times, useful for attribution only). Each worker builds the same adjacency independently.
   The existing shared-memory array design is correctly aimed at the dominant scaling problem.

6. **There is still a worthwhile pure-Python solver pass.** A combined POC removed an objective sum
   whose result is discarded, added exact `j_max == 0`/`1` fast paths, avoided allocating a Jaccard
   union, and cached held-route sort keys. An unprofiled 100k single-process run fell from 13.58 s to
   11.43 s (−15.8%) with exactly equal solutions. This is useful, but secondary to sharing static
   worker state.

If only one implementation batch is taken next, I would do the query orchestration fixes first. They
have a complete real-CLI before/after measurement, byte-identical output proof, and a meaningful
memory reduction. For setup, the largest-component POC deserves a focused integration story with a
full old/new graph diff because applying it cleanly requires controlling an osmnx-owned intermediate.

## Ranked findings

## 1. Query orchestration batch: measured end-to-end win

### 1a. Consume the operational graph during the query-side filter

`filter_trails` currently constructs an empty graph, copies all 131,745 nodes, then adds 327,492 kept
edges. The query has just deserialized and operationalized this graph and does not need the unfiltered
object after the difficulty-cap step. Collecting the 166 rejected edge identities and removing them
from that owned graph is equivalent.

Isolated r20 measurements:

| Variant | Wall | Result |
|---|---:|---|
| Build graph from kept nodes/edges | 4.98 s and 6.36 s in two POC runs | reference |
| Remove rejected edges in place | 0.309 s and 0.322 s | identical ordered edge list |

Real CLI stage:

| Baseline | POC |
|---:|---:|
| 4.19 s | 0.57 s |

The POC used the same `classify_highway` and `max_sac_rank` predicates, so it changed ownership, not
filter semantics. Rendering remains safe: every base edge reachable from a solver route is necessarily
in the routable graph, and super-edge expansion reads that same retained base set.

**Recommendation:** add an explicit consuming/internal variant rather than silently changing the
public pure function. The CLI can use `consume=True`; tests and external callers can keep the copying
default.

### 1b. Do not build the contracted graph twice

`contract_climbs` builds a full contracted graph whose connector edges still reference
`geometry` and `vertices_resampled`. `run_parallel_grasp` immediately calls `solver_graph_view`, which
builds a second graph solely to remove those two attributes.

The renderer does not read geometry from the contracted graph. It expands super-edges and connector
edges against `operational_graph`. The validator reads only metrics, SAC, descent, reuse tags, membership,
and node junction annotations.

Isolated r20 measurements:

| Operation | Wall |
|---|---:|
| Build `solver_graph_view` | 5.77 s / 6.09 s |
| Strip the same two attributes in place | 0.50 s / 0.55 s |
| Pickle rebuilt lean graph | 2.73–2.88 s, 73.1–73.2 MB |
| Pickle already-lean graph | 2.62–2.86 s, 73.1–73.2 MB |

The edge data of the stripped graph and rebuilt lean graph compared equal. In the real POC, stripping
was absorbed into the measured contraction stage:

| Baseline contraction | POC contraction + stripping |
|---:|---:|
| 7.76 s | 7.38 s |

This change also accounted for most of the peak-RSS reduction in the combined POC. It does **not**
remove child-side unpickling or adjacency construction; worker first-progress time stayed about 10–11 s.

**Recommendation:** keep `operational_graph` as the rendering graph and make the contracted graph lean
at construction time, or strip it in the CLI immediately afterward. Then skip `solver_graph_view` when
the graph advertises the lean contract.

### 1c. Compute validator graph invariants once

For every returned route, `_validate_edges` calls `non_exempt_base_segment_ids(graph)`, scanning all
327k contracted edges. With `n=10`, that is ten identical full-graph scans. Set validation separately
builds `base_segment_id_map(graph)`.

Isolated r20 result:

| Variant | Wall |
|---|---:|
| Current `validate`, 10 routes | 3.7705 s |
| Precompute non-exempt IDs | 0.3365 s |
| Precompute segment map | 0.4478 s |
| Validate all routes with those values | 0.0039 s |
| Output equality | exact |

Real CLI validate+render:

| Baseline | POC |
|---:|---:|
| 6.55 s | 2.87 s |

There is also a small duplicate `_route_metrics(edges)` call per route: `validate` builds the metrics,
then `_validate_edges` recomputes them for the slope check. Pass the already-computed metrics or average
into the private validator.

**Recommendation:** build a private `ValidationGraphContext` once in `validate` and pass it to every
route check and set check. Keep `validate_route`'s standalone API by letting it build a context when one
is not supplied.

## 2. Replace osmnx's double-traversal/copy largest-component step

osmnx 2.1.0's `largest_component` does this when a graph is disconnected:

1. traverse the graph with `is_weakly_connected`;
2. traverse it again with `weakly_connected_components`;
3. copy the large retained component through `nx.MultiDiGraph(G.subgraph(largest_cc))`.

The graph is a freshly created intermediate inside `graph_from_polygon`; steeproute does not need the
discarded object. The POC instead:

1. computes `max(weakly_connected_components(G), key=len)` once;
2. if necessary, removes nodes not in that set from the owned graph.

Warm-cache r20 timings:

| osmnx phase | Standard | Fast component |
|---|---:|---:|
| `_create_graph` | 20.42 s | 20.27 s |
| Truncate pass 1 | 19.47 s | 19.40 s |
| Largest component pass 1 | 25.40 s | 2.24 s |
| Simplify | 51.04 s | 48.71 s |
| Truncate pass 2 | 3.08 s | 4.35 s |
| Largest component pass 2 | 8.17 s | 0.54 s |
| Street count | 1.23 s | 1.21 s |
| **Total `osm_load`** | **131.94 s** | **99.10 s** |

Both real runs produced 131,793 nodes and 327,911 edges. The helper matched osmnx exactly—including
node order, edge order, node attrs, edge attrs, and graph attrs—on disconnected weak and strong test
graphs.

This does not change the largest-component policy and does not keep extra islands. It removes redundant
work only.

**Integration caution:** `graph_from_point` does not expose an “owned/in-place intermediate” hook.
Shipping this should not be an unscoped permanent monkeypatch. Options are:

- contribute the single-traversal improvement upstream to osmnx;
- implement a tightly version-guarded ingestion adapter around osmnx's lower-level calls;
- temporarily scope a patch under a lock, with an exact real-graph diff test and a pinned osmnx version.

The first option is cleanest but depends on release timing. A project-local adapter uses private APIs
and must be treated as such.

Before landing, run the story-14.5-style exact old/new diff on the cached r20 response: node IDs; edge
`(u,v,key)` IDs; relevant attrs; geometry coordinate sequences; and iteration order. The POC's equal
counts and helper proof are strong evidence, but not a substitute for that final full-content gate.

## 3. Consume other osmnx-owned intermediates

`truncate_graph_polygon` copies the entire graph before removing outside nodes, and `simplify_graph`
begins with another full copy. In the `graph_from_point` pipeline, these are owned temporaries.

A combined POC made truncation and simplification consume their inputs as well as using the fast
component helper. It preserved the real r20 node/edge counts and matched standard truncation +
simplification exactly on a disconnected bidirectional fixture.

Measured observations:

- first truncation: 19.4 s standard → 9.3 s consuming;
- peak RSS: 3.82 GB in the connectivity-only run → 2.70 GB in the all-consuming run;
- total: 111.70 s, slower than the 99.10 s connectivity-only run because simplification varied from
  48.7 s to 65.0 s and `_create_graph` from 20.3 s to 25.9 s.

The full-run timing is order/thermal/memory noisy, so do not claim another 10-second end-to-end win yet.
The first-truncation and memory results justify keeping this in the same focused osmnx story.

## 4. Remove setup elevation-sampling wrapper overhead

The stage-14.1 vectorized DEM lookup is good; the residual overhead is around it.

On the r20 prepared graph:

| Operation | Current shape | POC | Equality |
|---|---:|---:|---|
| Full `MultiDiGraph.copy()` | 5.386 s | avoid via consuming path | graph owned by setup |
| Gather + concatenate 2,862,827 coordinates per edge | 3.765 s | 1.513 s bulk `shapely.get_coordinates` | `np.array_equal` |

The user's complete elevation stage took 14.20 s. These measured components suggest roughly 7.6 s
of headroom, though only a real setup POC may claim the combined stage result.

**Recommendation:**

- add `inplace=False` (public default) / `consume=True` (or internal owned variant) to
  `sample_elevation`;
- collect geometry objects in edge order, call `shapely.get_coordinates(geoms, return_index=True)`
  once, derive offsets from the returned indices, and retain the current rasterio row/column logic;
- preserve the existing first-bad-edge error ordering.

This is simpler and lower risk than changing the DEM lookup itself.

## 5. Fuse setup polyline smoothing and resampling

Both stages already use the right flat-array strategy internally. The remaining waste is the boundary:

```text
flat source coordinates
  → smooth
  → build 327k-edge NetworkX/Shapely graph
  → gather that graph back into flat arrays
  → resample
  → build final graph
```

On the r20-shaped prepared graph under cProfile:

| Stage | Total profiled | `empty_like` + `_build_from_flat` |
|---|---:|---:|
| `smooth_polylines` | 9.07 s | 7.44 s |
| `resample_edges` | 9.40 s | 7.84 s |

The vectorized numeric work was not the bottleneck. A fused internal function can preserve the exact
current operation order:

1. `_collect_linestrings` once;
2. current smoothing array operations;
3. current resampling array operations using the smoothed array directly;
4. `_build_from_flat` once.

The public stage functions can remain for tests and callers; the setup orchestrator uses the fused
owned path. This should remove nearly the whole intermediate smoothing rebuild, but the acceptance
number must come from a real setup replay because the user's original resampling stage (31.31 s) was
slower than replaying resampling over the already-prepared graph.

## 6. Share or precompute solver static state

The worker graph and most solver initialization are pure functions of the contracted graph and a few
filter parameters, yet the current four-worker/four-round run repeats them extensively.

Repeated work includes:

- parent: `base_segment_id_map` for merges;
- each worker, each migration round: `base_segment_id_map`, `non_exempt_base_segment_ids`, sorted/junction
  node pool;
- each worker's first round: full adjacency build;
- validator: both graph-wide maps again (and currently non-exempt IDs per route).

The per-worker adjacency is cached across rounds, which already avoids the worst repeated round cost,
but the other static fields are rebuilt in every new `GraspSolver`.

Current r20 single-process cProfile, 100k iterations:

| Function | Profiled cumulative |
|---|---:|
| `_build_adjacency` | 12.36 s |
| `_construct_one`, 100k calls | 11.35 s |
| `GraspSolver.__init__` | 1.54 s |
| `base_segment_id_map` within init | 0.70 s |
| `non_exempt_base_segment_ids` within init | 0.58 s |

These are profiled times, not wall baselines, but the conclusion is unambiguous: at modest iteration
budgets, constructing identical object adjacency in every worker is as expensive as useful search.

### Recommended sequence

1. **Small step:** introduce a `SolverStaticContext` containing node pool, segment map, non-exempt IDs,
   and adjacency. Reuse it across migration rounds inside each worker. Return or reuse the parent's
   segment map for validation.
2. **Spawn-transfer step:** place the existing 73 MB pickled lean-graph blob in shared memory and pass
   only a descriptor to initializers. This avoids copying the same bytes through four spawn pipes and
   reduces transient memory, although each worker still unpickles its own graph.
3. **Structural fix:** implement the existing
   `steeproute-shared-memory-array-solver-design-2026-07-08.md`. Build one canonical CSR-style solver
   state in the parent and let workers read it directly. This is the only option that removes both:
   - O(workers) object-graph/adjacency construction;
   - O(workers × graph) steady worker memory.

Keep `--workers 1` on the current object path until array-path bit identity is proven. For N>1, compare
the array implementation against the current worker implementation over all quality-gate seeds and
the real fixture, not merely final objective totals.

## 7. Pure-Python solver-loop cleanup

A combined POC made four exact changes:

1. `_construct_one` no longer sums an objective that `run` discards before `_best_theta_prefix`
   recomputes the selected prefix objective;
2. `j_max == 0` uses `frozenset.isdisjoint` (any shared base segment means overlap);
3. `j_max == 1` skips overlap work (the current strict threshold can never classify overlap);
4. general Jaccard allocates only the intersection and computes union cardinality as
   `len(a) + len(b) - intersection`;
5. each held solution caches its deterministic sort key instead of re-sorting all edge IDs whenever
   `_worst_held` runs.

Unprofiled r20 single-process, 100k iterations:

| Current | Combined POC | Change | Output |
|---:|---:|---:|---|
| 13.580 s | 11.431 s | −15.8% | exact `list[Solution]` equality |

This is especially relevant because the quality-demo recommendation uses `j_max=0`. The general
Jaccard cardinality rewrite is mathematically identical and avoids constructing a union set.

Implement and benchmark these separately so the review records which change earns which gain. The
POC combined them to establish the ceiling, not to justify one opaque patch.

One correctness note discovered during review: Story 12.2's prose says builtin `sum(..., 0.0)` is a
naive left fold. On Python 3.13 it uses compensated float summation. `_best_theta_prefix`'s manual
cumulative arrays therefore are not generally bit-identical to `route_avg_gradient`; the final
canonical `_route_slope_ok(prefix)` gate is load-bearing. Do not remove it merely because the old
story says the values must agree.

## 8. Stop reconstructing cached geometry on the query path

The schema-v2 payload contains:

| Part | Shape / size |
|---|---:|
| Geometry coordinates | `(2,862,827, 2)`, 45,805,232 bytes |
| Geometry offsets | 327,659 entries, 1,310,636 bytes |

Warm measurements:

- pickle load: 1.50–2.02 s;
- validate payload and rebuild/reattach 327,658 LineStrings: 0.86–0.96 s.

Query stages 6–9 and rendering do not read the `geometry` attribute:

- stages 6–7 use `vertices_resampled`;
- contraction/solver/validator use metrics and tags;
- output expands routes against `vertices_resampled`.

The geometry is required during setup but is redundant in a prepared query cache once
`vertices_resampled` exists.

Options, in increasing scope:

1. let `read_entry` accept `with_geometry=False` for the query and skip bulk reconstruction;
2. split geometry arrays into a separate file so the query also avoids reading ~47 MB;
3. schema v3: omit post-stage-5 geometry entirely, after proving no supported consumer needs it.

Option 1 is an immediate ~0.9 s query-load win and saves LineString heap. Option 2/3 also reduce disk
I/O and cache size. Coordinate arrays currently live inline in the pickle, so merely skipping
reattachment does not skip their read.

This reverses the assumption behind Story 13.2—not because its measurement was wrong, but because the
query does not need reconstructed geometry at all.

## 9. Consume the graph during cache write

`_graph_to_payload` copies the whole graph before popping geometry. The setup CLI never uses the graph
after `write_entry`.

- r20 graph copy: 5.39 s;
- complete user cache-write stage: 7.70 s.

Add a consuming internal write path that extracts/pops geometry from the owned graph. Keep the public
copying behavior as default if external callers rely on purity. If schema v3 omits geometry, this
becomes even simpler.

## 10. Keep query elevation data flat across stages 6–7

`operationalize_graph` currently:

1. gathers graph values into arrays for Laplacian smoothing;
2. rebuilds every `vertices_resampled` tuple list;
3. scans and rebuilds the lists again for deadband;
4. converts every edge list back to arrays and concatenates for metrics.

The r20 cProfile attribution was:

| Function | Profiled cumulative |
|---|---:|
| `graph_smooth_elevation` | 7.56 s |
| `graph_deadband_elevation` | 2.31 s |
| `compute_edge_metrics` | 4.40 s |

The real combined stage was 10.09 s. A fused internal path can gather one flat `(lat, lon, elevation)`
representation, perform smoothing, deadband, and metrics over it, and rebuild tuple lists once for
rendering. Deadband remains sequential within each edge; fusion is about eliminating representation
round trips, not incorrectly vectorizing its hysteresis.

This is medium priority: likely a few seconds, but more delicate than the already-proven query
orchestration batch. Pin every metric and final vertex tuple before/after.

## Negative and deferred results

### Reducing osmnx `useful_tags_way`

I tested retaining only `highway`, `sac_scale`, `oneway`, and `junction` rather than osmnx's 15 default
way tags plus `sac_scale`.

The output was exactly equal for:

- node set;
- edge `(u,v,key)` set;
- `highway`, `sac_scale`, `osm_way_id`, and `oneway`;
- every geometry coordinate sequence.

It was not a demonstrated speedup:

| Full tags | Minimal tags |
|---:|---:|
| 131.94 s | 159.06 s |

The variants ran sequentially while both large graphs were resident, so order/memory/thermal effects
make the raw regression non-actionable. Phase timings showed no consistent reduction. Tag aggregation
is not the dominant simplification cost; do not prioritize this.

### More workers on the current object graph

Prior measurements already show throughput flattening and OOM at higher worker counts. The present
run reached 2.67 GB peak RSS with four workers at r20. Increasing workers before sharing static state
amplifies the wrong representation. Do the shared-state work first.

### Second-tier cached operational/contracted graphs

Story 13.3 already made a documented no-go decision because the query parameter product creates a
large invalidation/key surface. Nothing in this review overturns it. The flat cache-contract idea is
valuable when it improves every query, not as an unbounded per-parameter cache.

### Network concurrency

DEM fetching already uses four threads. This review made no extra IGN or Overpass requests and found
no evidence supporting higher server concurrency. Keep the existing polite limits.

### Quality-altering shortcuts

The following are not performance optimizations under this review's goal:

- lowering `theta`;
- changing `j_max`, difficulty, climb, descent, or junction constraints;
- retaining disconnected OSM islands;
- dropping the buffered/second truncation semantics;
- reducing iteration budget or changing RNG partitioning.

They change feasible routes, search effort, or quality and must not be mixed into an optimization
comparison.

## Suggested implementation plan

### Batch A — query orchestration, highest confidence

1. Add an owned/consuming query filter path.
2. Keep the contracted graph lean and eliminate `solver_graph_view` rebuilding.
3. Introduce one validation context per route set.
4. Pin byte-identical outputs on the exact r20 command and committed fixtures.
5. Measure CLI total, external process wall, and peak RSS.

Expected evidence target: reproduce approximately 80.0 → 67.3 s on r20. Do not promise the exact
number across machines.

### Batch B — setup owned-data/vectorization cleanup

1. Fuse smoothing → resampling.
2. Make elevation attachment consuming and use bulk geometry extraction.
3. Make cache write consuming.
4. Consider geometry-optional/schema-v3 cache storage.
5. Land these pipeline edits together to pay one content-hash/cache-regeneration event.

Measure a complete warm-response setup and a normal one-request setup. Report OSM, DEM, setup CPU,
query, and total separately.

### Batch C — osmnx component/copy adapter

1. Build an exact real-response old/new graph-diff harness first.
2. Apply the one-traversal in-place largest-component algorithm at the owned-intermediate boundary.
3. Evaluate consuming truncation/simplification in the same harness.
4. Version-guard private osmnx calls or upstream the change.
5. Re-run the full setup pipeline and query goldens; graph identity should make a rebake unnecessary.

### Batch D — solver static state and hot-loop cleanup

1. Cache static context across migration rounds.
2. Land the small pure-Python loop/tracker wins separately.
3. Prototype shared serialized input, then implement the existing shared-array worker design if r50
   remains the target.
4. Keep worker-1 exact behavior untouched.

## Expected combined headroom

Do not add every isolated saving mechanically: some work overlaps, and CPU/memory pressure changes the
timing of neighboring stages. The defensible statements are:

- **Query:** 12.69 s CLI-wall reduction is already demonstrated as one combined, byte-identical POC.
- **Warm OSM ingestion:** 32.85 s reduction is demonstrated for the component POC.
- **Elevation wrapper:** 7.6 s of components is measured, but not yet demonstrated as a combined stage.
- **Smoothing/resampling fusion:** intermediate rebuild cost is measured, but not yet demonstrated on
  the original pre-resample r20 graph.
- **Solver loop:** 2.15 s per 100k single-process run is demonstrated for the combined micro-POC;
  four-worker end-to-end effect is not yet measured.
- **Shared worker arrays:** potentially the largest solver startup/memory win, but still a design,
  not a measured implementation.

The current 299.28 s setup includes 82.87 s of DEM transfer and some Overpass wait, so even perfect
CPU removal cannot make the first preparation instantaneous. The current 80.02 s query has no such
network floor and therefore benefits more directly from the proven orchestration changes.

## Bottom line

The code is already well optimized numerically. The next pass should focus less on additional NumPy
math and more on ownership:

- do not copy a graph that the caller has finished with;
- do not rebuild a graph merely to remove two attributes;
- do not derive immutable graph-wide sets per route or per migration round;
- do not reconstruct cache data no query consumer reads;
- do not give every worker a private object graph when they all read the same state.

Those ownership changes are where the current measurements show the next substantial speedups, and
the strongest ones preserve output byte-for-byte.
